  <name>Task 1: Create pipeline orchestrator with tiered execution</name>
  <files>
    pipeline/main.py

  <action>
    Create the main pipeline orchestrator that runs all ingestors with tiered failure handling:

    **pipeline/main.py:**

    - Import all ingestors: rba_data, abs_data, corelogic_scraper, nab_scraper
    - Import sys, json, datetime

    - Define source tiers:
      * CRITICAL_SOURCES = [
          ('RBA Cash Rate', rba_data),
          ('ABS Economic Data', abs_data),
        ]
      * OPTIONAL_SOURCES = [
          ('CoreLogic Housing', corelogic_scraper),
          ('NAB Capacity', nab_scraper),
        ]

    - `run_pipeline() -> dict`:
      * Initialize results dict: {'run_date': ISO timestamp, 'critical': {}, 'optional': {}, 'status': 'pending'}
      * **Phase 1 - Critical sources (fail fast):**
        - For each (name, module) in CRITICAL_SOURCES:
          * Try: result = module.fetch_and_save()
          * Store in results['critical'][name] = result
          * On exception: print "CRITICAL FAILURE: {name} failed: {error}", set results['status'] = 'failed', sys.exit(1)
      * **Phase 2 - Optional sources (graceful degradation):**
        - For each (name, module) in OPTIONAL_SOURCES:
          * Try: result = module.fetch_and_save()
          * Store in results['optional'][name] = result
          * On exception: print "WARNING: {name} failed: {error}", store error in results
      * Determine final status:
        - All succeeded: results['status'] = 'success'
        - Optional failures: results['status'] = 'partial'
      * Print summary: total sources, successes, failures
      * Return results

    - `if __name__ == '__main__':`:
      * results = run_pipeline()
      * Print JSON summary to stdout
      * Exit code: 0 for success/partial, 1 for critical failure (handled by sys.exit in run_pipeline)

    **Exit code convention (from research):**
    - 0 = all sources succeeded
    - 1 = critical source failed (pipeline failed)
    - 2 = optional source(s) failed (partial success) -- use sys.exit(2) if any optional failures

    The exit code distinction lets GitHub Actions treat partial success differently from total failure.
  </action>
  <verify>
    Run: `python pipeline/main.py` -- should execute all ingestors in order, print summary
    Verify critical-first ordering: critical sources should run before optional in output
    Verify partial failure handling: if an optional source fails, pipeline still completes with exit code 2
    Verify output: JSON summary printed to stdout showing all source results
  </verify>
  <done>
    Pipeline orchestrator runs all sources in tiered order. Critical failures abort immediately (exit 1). Optional failures degrade gracefully (exit 2). Full success exits 0. PIPE-11 satisfied.

<task type="auto">
  <name>Task 2: Create GitHub Actions workflows</name>
  <files>
    .github/workflows/weekly-pipeline.yml

  <action>
    Create two GitHub Actions workflow files:

    **1. .github/workflows/weekly-pipeline.yml:**

    ```yaml
    name: Weekly Data Pipeline

    on:
      schedule:
        - cron: '7 2 * * 1'       # Monday 2:07 AM UTC (off-peak, avoids :00)
      workflow_dispatch:            # Manual trigger support

    permissions:
      contents: write              # Required for git-auto-commit-action

    jobs:
      ingest:
        runs-on: ubuntu-latest
        steps:
          - name: Checkout repository
            uses: actions/checkout@v6

          - name: Set up Python
            uses: actions/setup-python@v6
            with:
              python-version: '3.11'
              cache: 'pip'

          - name: Install dependencies
            run: pip install -r requirements.txt

          - name: Run data pipeline
            run: python pipeline/main.py

          - name: Commit updated data
            uses: stefanzweifel/git-auto-commit-action@v7
            with:
              commit_message: 'data: weekly pipeline update [skip ci]'
              file_pattern: 'data/*.csv'
              commit_user_name: 'github-actions[bot]'
              commit_user_email: 'github-actions[bot]@users.noreply.github.com'
    ```

    **2. .github/workflows/daily-asx-futures.yml:**

    ```yaml
    name: Daily ASX Futures Scraper

    on:
      schedule:
        - cron: '23 6 * * 1-5'    # Weekdays 6:23 AM UTC (after AU market close)
      workflow_dispatch:            # Manual trigger support

    permissions:
      contents: write

    jobs:
      scrape-futures:
        runs-on: ubuntu-latest
        steps:
          - name: Checkout repository
            uses: actions/checkout@v6

          - name: Set up Python
            uses: actions/setup-python@v6
            with:
              python-version: '3.11'
              cache: 'pip'

          - name: Install dependencies
            run: pip install -r requirements.txt

          - name: Scrape ASX futures
            run: python -m pipeline.ingest.asx_futures_scraper

          - name: Commit updated data
            uses: stefanzweifel/git-auto-commit-action@v7
            with:
              commit_message: 'data: daily ASX futures update [skip ci]'
              file_pattern: 'data/asx_futures.csv'
              commit_user_name: 'github-actions[bot]'
              commit_user_email: 'github-actions[bot]@users.noreply.github.com'
    ```

    **Key design decisions:**
    - Cron times at odd minutes (:07, :23) to avoid GitHub Actions load spikes at :00
    - Weekly pipeline runs Monday for consistency with decision in CONTEXT.md
    - ASX futures runs weekdays only (markets closed weekends)
    - ASX futures at 6:23 UTC = ~4:23 PM AEST (after AU market close, data is fresh)
    - Both use `[skip ci]` in commit message to prevent workflow trigger loops
    - Both use GITHUB_TOKEN (built-in) not PAT -- prevents infinite loops
    - pip caching enabled via actions/setup-python cache: 'pip'
    - permissions: contents: write is required for git-auto-commit-action
  </action>
