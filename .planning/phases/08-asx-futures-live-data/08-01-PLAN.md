Insert a new step in `.github/workflows/daily-asx-futures.yml` AFTER the "Regenerate status.json" step and BEFORE the "Commit updated data" step:

```yaml
      - name: Verify ASX data freshness
        run: |
          python3 -c "
          import pandas as pd
          from datetime import datetime, timedelta
          df = pd.read_csv('data/asx_futures.csv')
          df['date'] = pd.to_datetime(df['date'])
          latest = df['date'].max()
          threshold = datetime.now() - timedelta(days=7)
          if latest < threshold:
              print(f'WARNING: Latest ASX data is {latest.date()}, older than 7 days')
              exit(1)
          print(f'OK: Latest ASX data is {latest.date()}')
          "
```

This step:
- Reads `data/asx_futures.csv` (the path in CI, not `DATA_DIR` which is relative to the Python package)
- Checks if the maximum `date` column value is within the last 7 days
- Exits with code 1 if stale (this will fail the CI step but the commit step still runs because of how the workflow is structured — the scraper already ran and data was written)
- Important: The freshness check uses `exit(1)` which will mark the step as failed but won't prevent the commit step from running since the commit step is independent. If we want the commit to still happen even when freshness fails, add `continue-on-error: true` to this step.

Add `continue-on-error: true` to the freshness step so stale data still gets committed (the warning is sufficient — we don't want to block data commits just because the endpoint was intermittently down):
