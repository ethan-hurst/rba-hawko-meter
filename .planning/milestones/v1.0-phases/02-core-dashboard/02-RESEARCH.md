# Phase 2: Core Dashboard - Research

**Researched:** 2026-02-04
**Domain:** Static dashboard development with vanilla JavaScript, Plotly.js, and Tailwind CSS
**Confidence:** HIGH

## Summary

This phase builds a static HTML/JavaScript dashboard hosted on Netlify with a dark finance-themed aesthetic. The technical foundation is well-established: Plotly.js for interactive time series charts, Tailwind CSS for responsive dark mode styling, and vanilla JavaScript for countdown timers and data loading.

The research confirms that this tech stack is optimal for static dashboards. Plotly.js is the standard for interactive financial charts with 37k+ GitHub stars, Tailwind CSS provides production-ready dark mode support with minimal configuration, and vanilla JavaScript patterns eliminate framework overhead while maintaining clean architecture through ES6 modules.

Key findings: Plotly.js requires specific configuration for responsive mobile behavior, dark mode implementation needs careful color contrast planning (4.5:1 minimum), and countdown timers must properly clean up setInterval references to prevent memory leaks.

**Primary recommendation:** Use Plotly.js CDN with the basic bundle (~1MB) for faster loading, implement Tailwind's class-based dark mode with localStorage persistence, and structure vanilla JS using ES6 modules with IIFE patterns for encapsulation.

### Pattern 1: ES6 Module Pattern with IIFE

**What:** Use ES6 modules for code organization with IIFE (Immediately-Invoked Function Expressions) for encapsulation
**When to use:** Vanilla JavaScript projects without bundlers that need private scope and public APIs

**Example:**
```javascript
// js/countdown.js
const CountdownModule = (() => {
  let intervalId = null;

  const calculateTimeRemaining = (targetDate) => {
    const now = new Date();
    const difference = targetDate - now;

    return {
      days: Math.floor(difference / (1000 * 60 * 60 * 24)),
      hours: Math.floor((difference / (1000 * 60 * 60)) % 24),
      minutes: Math.floor((difference / (1000 * 60)) % 60),
      seconds: Math.floor((difference / 1000) % 60),
      isExpired: difference < 0
    };
  };

  const startCountdown = (targetDate, callback) => {
    // Clear any existing interval to prevent memory leaks
    if (intervalId) clearInterval(intervalId);

    intervalId = setInterval(() => {
      const timeRemaining = calculateTimeRemaining(targetDate);
      callback(timeRemaining);

      if (timeRemaining.isExpired) {
        stopCountdown();
      }
    }, 1000);

    // Immediate first call
    callback(calculateTimeRemaining(targetDate));
  };

  const stopCountdown = () => {
    if (intervalId) {
      clearInterval(intervalId);
      intervalId = null;
    }
  };

  // Public API
  return {
    start: startCountdown,
    stop: stopCountdown
  };
})();
```

### Pattern 5: Safe DOM Updates (XSS Prevention)

**What:** Update DOM content safely without innerHTML to prevent XSS attacks
**When to use:** All dynamic content updates, especially with user-provided or external data

**Example:**
```javascript
// js/data.js
const DataModule = (() => {
  const cache = new Map();

  const fetchData = async (url) => {
    // Check cache first
    if (cache.has(url)) {
      return cache.get(url);
    }

    try {
      const response = await fetch(url);

      if (!response.ok) {
        throw new Error(`HTTP error! status: ${response.status}`);
      }

      const data = await response.json();
      cache.set(url, data);
      return data;
    } catch (error) {
      console.error('Failed to fetch data:', error);
      throw new Error(`Failed to load data from ${url}`);
    }
  };

  const displayError = (containerId, message) => {
    const container = document.getElementById(containerId);

    // Clear existing content
    container.textContent = '';

    // Create error elements safely
    const errorDiv = document.createElement('div');
    errorDiv.className = 'bg-red-900/20 border border-red-500 text-red-200 px-4 py-3 rounded';

    const titleP = document.createElement('p');
    titleP.className = 'font-bold';
    titleP.textContent = 'Error Loading Data';

    const messageP = document.createElement('p');
    messageP.className = 'text-sm';
    messageP.textContent = message;

    errorDiv.appendChild(titleP);
    errorDiv.appendChild(messageP);
    container.appendChild(errorDiv);
  };

  return {
    fetch: fetchData,
    showError: displayError
  };
})();
```

### Anti-Patterns to Avoid

- **Dynamic class name construction in Tailwind:** `class="text-${color}-500"` won't work with JIT. Use complete class names or safelist dynamic classes.
- **Not clearing setInterval:** Always store interval IDs and clear them in cleanup/unmount logic to prevent memory leaks.
- **Relying on Plotly auto-responsiveness for container changes:** `responsive: true` only works on window resize, not parent container resize. Manually call `Plotly.Plots.resize()` when containers change.
- **Using Math for date manipulation:** Use native Date API or date-fns library for timezone-safe date calculations, especially for AEST/AEDT.
- **Putting large data in HTML:** Load data from separate JSON files to enable caching and keep HTML lightweight.
- **Using innerHTML with dynamic content:** Prefer textContent for text or safe DOM methods (createElement, appendChild) to prevent XSS vulnerabilities.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Time series chart with zoom/pan | Custom canvas/SVG drawing | Plotly.js | Handles date axes, hover interactions, annotations, touch gestures, and accessibility out of the box |

| Responsive chart resizing | Custom resize observers + debouncing | Plotly config.responsive | Already optimized with performance throttling and cross-browser compatibility |
| Date formatting | String manipulation | Intl.DateTimeFormat API | Handles locales, timezones (AEST/AEDT), and formats without external libraries |

**Key insight:** Financial dashboards have well-established patterns. Plotly.js and Tailwind CSS eliminate 90% of the custom code you'd otherwise write for charts, dark mode, and responsive behavior. Focus implementation effort on data transformation and business logic, not reinventing charting primitives.

## Common Pitfalls

### Pitfall 1: Plotly Charts Not Responsive on Mobile

**What goes wrong:** Charts render at fixed width or don't respond to screen orientation changes despite setting `responsive: true`

**Why it happens:** Plotly's `responsive` config only triggers on window resize events, not when CSS media queries change or parent containers resize. Also, the chart container must have explicit dimensions or use flexbox/grid properly.

**How to avoid:**
- Set chart container to `width: 100%; height: 100vh;` or use Tailwind `w-full h-96`
- Always include `config: { responsive: true }` in Plotly.newPlot call
- For orientation changes, manually call `Plotly.Plots.resize('myDiv')` after `window.matchMedia` orientation change events
- Test on actual mobile devices, not just browser dev tools (touch events behave differently)

**Warning signs:**
- Horizontal scrolling on mobile viewports
- Charts that are too wide or too narrow
- Fixed pixel widths in layout config (avoid `layout.width = 800`)

**Source:** [Plotly.js responsive/fluid layouts](https://plotly.com/javascript/responsive-fluid-layout/), [GitHub Issue #3984](https://github.com/plotly/plotly.js/issues/3984)

### Pitfall 2: setInterval Memory Leaks in Countdown Timer

**What goes wrong:** Memory usage grows over time, especially in single-page apps or when navigating away and back to the dashboard

**Why it happens:** `setInterval` creates a closure that references DOM elements and data. If the interval isn't cleared before the page/component is destroyed, those references prevent garbage collection.

**How to avoid:**
- Always store the interval ID: `let intervalId = setInterval(...)`
- Clear it before creating a new one: `if (intervalId) clearInterval(intervalId)`
- Clear it on page unload: `window.addEventListener('beforeunload', () => clearInterval(intervalId))`
- Don't reference large objects in the interval callback unnecessarily

**Warning signs:**
- Increasing memory usage in Chrome DevTools Performance Monitor
- Multiple timers running simultaneously (check with `console.log` in callback)
- Browser tab becomes sluggish after leaving page open for hours

**Source:** [LogRocket - Memory Leaks](https://blog.logrocket.com/escape-memory-leaks-javascript/), [Best Practices Medium](https://mdjamilkashemporosh.medium.com/best-practices-for-avoiding-memory-leaks-in-javascript-1fb24827914b)

### Pitfall 3: Insufficient Color Contrast in Dark Mode

**What goes wrong:** Text or chart elements are difficult to read on dark backgrounds, especially for users with visual impairments or on poor quality screens

**Why it happens:** Developers often choose aesthetically pleasing colors without testing contrast ratios. WCAG requires 4.5:1 for normal text, 7:1 for AAA compliance.

**How to avoid:**
- Use a contrast checker (WebAIM, Chrome DevTools) for all text/background combinations
- For finance dashboards, maintain at least 4.5:1 contrast ratio
- Test Plotly chart colors: default `plotly_dark` template may need custom colors
- Avoid pure black (#000000) backgrounds; use softer blacks like #0a0a0a or #1a1a1a
- For gauge colors: ensure red/amber/green are distinguishable even for colorblind users (test with simulators)

**Warning signs:**
- Gray text on dark gray backgrounds
- Chart grid lines that are invisible
- Annotations with poor visibility

**Source:** [Phoenix Strategy Group - Financial Dashboard Colors](https://www.phoenixstrategy.group/blog/best-color-palettes-for-financial-dashboards), [Muzli Dashboard Design 2026](https://muz.li/blog/best-dashboard-design-examples-inspirations-for-2026/)

### Pitfall 4: Tailwind JIT Not Detecting Dynamic Classes

**What goes wrong:** Classes work in development but are missing in production build, causing unstyled elements

**Why it happens:** Tailwind's JIT compiler scans source files for complete class names at build time. Dynamic class construction (`text-${color}-500`) or classes generated at runtime won't be detected.

**How to avoid:**
- Use complete class names: `text-red-500` not `text-${color}-500`
- If dynamic classes are necessary, use the `safelist` option in Tailwind config
- Keep class names as plain strings in templates, not concatenated
- In Tailwind v4, use `@utility` for custom utilities that should be purged

**Warning signs:**
- Styles work with Tailwind CDN but fail with production build
- Intermittent styling issues depending on which pages were visited
- Missing classes in production CSS file inspection

**Source:** [Tailwind Optimizing for Production](https://v2.tailwindcss.com/docs/optimizing-for-production), [GitHub Discussion #7568](https://github.com/tailwindlabs/tailwindcss/discussions/7568)

### Pitfall 5: Date/Time Confusion with AEST/AEDT

**What goes wrong:** Countdown shows wrong time, or meeting dates display as day before/after expected

**Why it happens:** RBA meeting times are in Australian Eastern Time (AEST/AEDT) which has daylight saving, but user's browser may be in different timezone. JavaScript Date objects use UTC internally.

**How to avoid:**
- Store meeting dates in ISO 8601 format with timezone: `2026-02-03T14:30:00+11:00`
- Use `Intl.DateTimeFormat` with `timeZone: 'Australia/Sydney'` for display
- Don't use string parsing like `new Date('2026-02-03')` (treats as UTC midnight)
- For countdown calculations, convert both dates to UTC milliseconds
- Show timezone in UI: "2:30pm AEST" not just "2:30pm"

**Warning signs:**
- Countdown is off by several hours
- "Meeting today" banner shows on wrong day
- Different behavior between Sydney users and international users

**Source:** [MDN Date Reference](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Date), [Intl.DateTimeFormat](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Intl/DateTimeFormat)

### Countdown Timer Implementation

```javascript
// Source: Best practices from LogRocket, Medium articles
const initCountdown = (targetDate) => {
  const countdownEl = document.getElementById('countdown');
  let intervalId = null;

  const updateDisplay = (time) => {
    // Clear existing content
    countdownEl.textContent = '';

    const containerDiv = document.createElement('div');

    if (time.isExpired) {
      containerDiv.className = 'bg-amber-900/20 border border-amber-500 px-6 py-4 rounded-lg';

      const messageP = document.createElement('p');
      messageP.className = 'text-amber-200 font-bold text-xl';
      messageP.textContent = 'RBA meeting TODAY — decision expected 2:30pm AEST';

      containerDiv.appendChild(messageP);
    } else {
      containerDiv.className = 'text-gray-300';

      const text = document.createTextNode('Next meeting in ');
      const timeSpan = document.createElement('span');
      timeSpan.className = 'font-bold text-white';
      timeSpan.textContent = `${time.days}d ${time.hours}h ${time.minutes}m ${time.seconds}s`;

      containerDiv.appendChild(text);
      containerDiv.appendChild(timeSpan);
    }

    countdownEl.appendChild(containerDiv);
  };

  const calculateTime = (target) => {
    const now = new Date();
    const diff = target - now;

    return {
      days: Math.floor(diff / (1000 * 60 * 60 * 24)),
      hours: Math.floor((diff / (1000 * 60 * 60)) % 24),
      minutes: Math.floor((diff / (1000 * 60)) % 60),
      seconds: Math.floor((diff / 1000) % 60),
      isExpired: diff < 0
    };
  };

  // Clear any existing interval
  if (intervalId) clearInterval(intervalId);

  // Start new countdown
  intervalId = setInterval(() => {
    updateDisplay(calculateTime(targetDate));
  }, 1000);

  // Initial display
  updateDisplay(calculateTime(targetDate));

  // Cleanup on page unload
  window.addEventListener('beforeunload', () => {
    if (intervalId) clearInterval(intervalId);
  });
};
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| PurgeCSS for Tailwind | JIT (Just-In-Time) compilation | Tailwind v3 (2021) | Eliminates build-time purging, generates only used classes on-demand, faster builds |
| Plotly via npm with bundler | Plotly CDN with partial bundles | Ongoing (2020+) | Static sites can avoid npm entirely; basic bundle reduces from 3MB to <1MB |
| prefers-color-scheme only | Class-based dark mode with localStorage | Tailwind v2.1+ (2021) | Users can manually toggle dark mode regardless of system preference |
| jQuery for DOM manipulation | Vanilla JavaScript with ES6 modules | 2015-2020 transition | Native modules, better performance, zero dependencies for simple sites |
| Custom date formatting | Intl.DateTimeFormat API | Widespread support 2019+ | Native timezone handling, no external libraries like moment.js needed |
| innerHTML for dynamic content | textContent + safe DOM methods | Ongoing security best practice | Prevents XSS vulnerabilities in dynamic content |

**Deprecated/outdated:**
- **Moment.js:** Declared as "legacy project" in 2020. Use native Intl or date-fns instead.
- **Tailwind "purge" config:** Replaced by "content" in v3. JIT mode makes purging automatic.
- **Plotly Python offline mode:** Server-side rendering less common now; CDN + client-side is standard for dashboards.
- **jQuery:** Unnecessary for modern browsers with native ES6+ support.
- **innerHTML for untrusted content:** Security risk; use textContent for text or createElement for structured content.

## Open Questions

Things that couldn't be fully resolved:

1. **Plotly.js bundle size optimization**
   - What we know: Full bundle is 3MB, basic bundle is ~1MB, custom bundles possible
   - What's unclear: Whether the basic bundle includes all features needed (scatter, line, annotations, shapes)
   - Recommendation: Start with CDN basic bundle, test if annotations work. If not, use full bundle (3MB is acceptable for dashboard)

2. **RBA meeting date data source**
   - What we know: Meeting dates need to be in a JSON file for countdown
   - What's unclear: Whether backend (Phase 1) generates this or it's manually maintained
   - Recommendation: Assume `data/meetings.json` exists with structure `{ "nextMeetingDate": "2026-03-03T14:30:00+11:00" }`. Document dependency in planning.

3. **Timezone handling for international users**
   - What we know: RBA meetings are AEST/AEDT, need to show correct timezone
   - What's unclear: Whether to show meeting time in user's local timezone or always in AEST
   - Recommendation: Always display AEST (canonical timezone for RBA), but show countdown in user's local time. Add "(Sydney time)" label for clarity.

4. **Accessibility testing requirements**
   - What we know: Color contrast must be 4.5:1 minimum
   - What's unclear: Whether keyboard navigation, screen readers, ARIA labels are in scope for v1
   - Recommendation: Implement basic color contrast and semantic HTML. Defer advanced a11y (ARIA, screen reader testing) to post-v1 unless COMP-03 disclaimer requires it.
